# RELEASE.md — 改良・公開・マーケティング実行規約

このファイルは `GOAL.md`(Phase 0〜7 完了済み)の続きとして、Claude Code が
`/loop` の中で参照すること。`PRINCIPLES.md` / `SPEC.md` / `PROGRESS.md` は
引き続き有効。本書は追加フェーズ(Phase A〜F)を定義し、あわせて
`GOAL.md` の一部条件を明示的に改訂する。

## GOAL.md からの変更点(重要)

1. **エスカレーション条件4(リポジトリの public 化)を解除する。**
   人間からの明示的な承認により、public 化の最終アクションを含め、
   本書の Phase A〜F は Claude Code が自律的に実行してよい。
   ただし Phase A(README/install 改修)完了 → 自己検証済み、
   の状態を満たした後にのみ Phase C(公開)を実行すること。
2. **匿名性に関する既知のリスクを記録する。**
   GitHub アカウントに使用するメールアドレスには、実名に近い文字列
   (`chibahiroaki`)が含まれることを開発者に確認済みであり、
   「実名流出は許容する」という判断が下されている。これは
   `GOAL.md` エスカレーション条件2に該当する事項だが、**既に人間が
   判断済みの受容リスク**として扱い、これ以上の匿名化措置(履歴書き換え
   等)は不要。Claude Code はこの点について再度確認を求めなくてよい。
3. **GitHub author 名は `Ark-Karim` に統一する。** README・NOTICE・
   マーケティング文章など、対外的に表示される名前はすべてこれに揃える。

4. umeboshiKaname notetype について、現状あなたは、build 後のファイルのみ含めているが、コミュニティ開発を視野に入れているため、src file を含める必要がある。

## 前提(PROGRESS.md より)

- Phase 0〜7 は完了済み。secret scan CI・自己検証も完了と報告されている。
- ただし、README の内容(生成方式・コスト構成)は今回の方針転換により

## Phase A: README・セットアップ体験の改修(公開前の必須ゲート)

**Status**: Not Started

以下を反映すること。優先順位はこの順番。

### A-1. カード生成: Claude subscription をメインに

- Claude Code CLI がユーザー自身の Claude Pro / Max 等のサブスクリプション
  で動作することを前提とし、これをメインの生成方法として README のトップに
  記載する。**別途 Anthropic API key の発行は不要**、という理解で
  README を書く。ただし、実際の Claude Code の認証方式・料金体系・
  利用上限については記憶に頼らず、必ず `docs.claude.com` の最新情報を
  確認してから記載すること(仕様が変わっている可能性があるため)。
- DeepSeek は「サブスクリプション費用を払いたくない/持っていない人向けの
  代替オプション」として、メインの手順の下に補足的に記載する。

### A-2. 文字起こし: ローカル Whisper をデフォルトに

- デフォルトはローカル Whisper(英語想定)とする。
- 日本語など非英語の場合は、オプトインで外部サービス(OpenRouter /
  Fireworks 等)を設定すれば使える、という位置づけにする。
- OpenRouter / Fireworks のモデルラインナップは変化が早いため、
  README には特定モデル名をハードコードしすぎず、
  「文字起こしバックエンド」を抽象化した設定(後述 Phase B の
  モジュール設計と連動)として案内する。
- 将来的に文字起こしサービスをプラグイン的に切り替えられる設計を
  見据えること(Phase B 参照)。

### A-3. コストの透明性

- DeepSeek 経由: 90分の日本語講義で約 $0.30 という実測値を明記する。
- Claude subscription 経由: 「サブスクリプション料金の範囲内で追加コスト
  なし」が基本メッセージだが、ユーザーがイメージを持てるように、
  実際どの程度の使用量(トークン数・リクエスト数のオーダー)に相当するかを
  概算で示す。この際、Claude Code の現在の利用上限・料金体系を
  `docs.claude.com` で確認し、記憶による古い数値を使わないこと。
  不確実な場合は「最新の詳細は docs.claude.com を参照」とリンクする。
- ローカル Whisper 使用時: 追加コストなし(電力・PC性能以外)と明記する。

### A-4. install / セットアップの再設計

- 上記 A-1〜A-3 の変更に合わせて `install.sh` と `docs/SETUP.md` を更新する。
- README 本体は A4 一枚の制約を維持したまま、「デフォルト(ローカル
  Whisper + Claude subscription)だけで完結する最短経路」を最初に示し、
  オプション(DeepSeek、外部文字起こしサービス)は折りたたむ・別セクション
  にするなど、情報量を制御すること。

### A-5. 自己検証の再実施

- README改修後、Claude Code は「事情を知らない第三者」のつもりで、
  デフォルト経路(ローカル Whisper + Claude subscription)のみを使って
  再現できるかを自己検証する。
- 検証結果を `PROGRESS.md` に記録する。ここが未完了のまま Phase C
  (公開)に進んではならない。

## Phase B: モジュール化(Anki 本体レベルの設計を目標に)

**Status**: Not Started
**Note**: 開発者判断により、このフェーズは Phase C(公開)の必須前提条件と
された(「中途半端な公開はファーストインプレッションを損なう」という理由)。
規模が大きく、Phase A よりも時間がかかる可能性が高いことを認識した上で、
それでも公開前に完了させる方針である。Claude Code は工数を惜しまず、
段階的に(境界定義 → 機構実装 → ドキュメント → サンプル → 既存実装の
移行、の順で)着実に進めること。

### 方針

Anki 本体(アプリケーション)が持つような、明確なモジュール境界と
アドオン(プラグイン)機構を目標とする。**開発者の判断により、これは
「中途半端な状態での公開はファーストインプレッションを損なう」という
理由から、公開前(Phase C 以前)の必須ゲートとして扱う。** 規模の大きい
作業だが、Phase C(公開)より優先する。

### 今回のスコープ(公開前に完了させること)

1. **モジュール境界の定義**: 少なくとも以下を独立したモジュール/
   インターフェースとして切り出す
   - 文字起こしバックエンド(ローカル Whisper / OpenRouter / Fireworks
     / 将来の追加サービスを共通インターフェースで切り替え可能に)
   - カード生成バックエンド(Claude Code / DeepSeek / 将来の追加モデル)
   - Anki import バックエンド(AnkiConnect / .apkg エクスポート)
   - notetype 描画(Umeboshi-Kaname / 将来の別テンプレート)
2. **プラグイン機構そのものの実装**: 上記モジュールを、コア本体を
   書き換えずに追加・入れ替えできる形にする(例: 決まったディレクトリに
   規約に沿ったファイルを置けば自動検出される、共通インターフェース/
   基底クラスを継承すれば新しいバックエンドとして登録される、など)。
   Anki のアドオンマネージャーに相当する「読み込み・登録の仕組み」を
   実装すること。
3. **`docs/ARCHITECTURE.md` の作成**: 各モジュールの役割・入出力・
   拡張ポイントを、**エンジニアでない人でも図や具体例を見れば大枠を
   理解できるレベル**で説明する。
4. **`docs/PLUGIN_DEVELOPMENT.md`(プラグインの作り方)の作成**:
   第三者が新しい文字起こしバックエンド・生成バックエンド・import
   バックエンド・notetype を自作できるよう、
   - 実装すべきインターフェース/規約
   - 最小のサンプルコード(Hello World 的な最小プラグイン)
   - 動作確認・デバッグの方法
   - 公式リポジトリへの取り込みを希望する場合の連絡・PRの流れ
   を具体的に記載する。
5. **サンプルプラグインの同梱**: 上記ドキュメントの説明が机上の空論に
   ならないよう、`plugins/examples/` 等に実際に動く最小サンプル
   プラグイン(例: ダミーの文字起こしバックエンドなど)を最低1つ用意し、
   ドキュメントの手順通りに動くことを自己検証する。
6. 既存の `skills/` 配下の実装を、上記のプラグイン機構に沿う形で
   実際にリファクタリングする(既存の文字起こし・生成・import の
   実装が、新しいプラグイン機構経由で動くようにする)。

### スコープ外(このリリースでは着手しない)

- サードパーティ製プラグインを一元的に配布・検索できるレジストリ/
  マーケットプレイス(例: Anki の AnkiWeb のような中央配布基盤)。
  これは公開後、実際にコントリビューターが現れてから検討する
  ロードマップ項目として `PROGRESS.md` に記録する。
