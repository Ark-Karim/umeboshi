# GOAL.md — /loop 実行規約

このファイルは `/loop` の各反復で最初に読み込むこと。
判断の思想的根拠は `PRINCIPLES.md`、技術的な制約は `SPEC.md`、
背景は `PROJECT.md` を参照すること。

## ミッション(1文で)

このリポジトリを、「README を読んだだけで技術者でない人が迷わず動かせる、
匿名の Apache 2.0 OSS プロジェクト」として公開できる状態にする。

## 意思決定の原則

- 判断に迷ったら、まず `PRINCIPLES.md` を参照する。
- `PRINCIPLES.md` に明記されていない判断は、Claude 自身の一般的な良識・
  訓練された判断力に基づいて決めてよい。**開発者(人間)の個人的な好みを
  推測して忖度する必要はない。** 開発者の意図は `PRINCIPLES.md` を通じて
  のみ反映されている、という前提で動くこと。
- 「これで合っているか」を都度人間に確認する必要はない。以下の
  「エスカレーション条件」に該当しない限り、自律的に進めて良い。

## 裁量の範囲(確認不要で進めてよいこと)

- ファイル構成・リファクタリング・実装方式の選定
- README / ドキュメントの文言、構成、分量の調整
- サンプル教材の選定(著作権フリーのものに限る)
- 音声処理方式(OpenRouter Whisper vs ローカル Whisper)の比較・選定
- Anki import 方式(AnkiConnect / .apkg / OS分岐)の比較・選定
- テストコード、CI設定、install スクリプトの作成
- コミット、通常の push
- 依存ライブラリの選定・追加

## エスカレーション条件(必ず人間に確認・報告してから進めること)

以下に該当する場合は、作業を止めて `PROGRESS.md` に状況を明記し、
人間の判断を仰ぐこと。**自律的に処理を完了させない。**

1. **git 履歴や現在のコードに API キー・認証情報の漏出が見つかった場合**
   (`SPEC.md` セクション5)。特に、履歴の書き換え(force-push)を要する
   修正は必ず人間の確認を経ること。
2. **個人を特定しうる情報**(実名、所属機関、特定できる音声データなど)が
   見つかった場合。匿名性が崩れる可能性があるため。
3. **ライセンス・著作権上の判断が曖昧なケース**(サンプル教材の
   ライセンス表記が不明瞭、AnKing 由来コードの範囲が不明確、など)。
4. **リポジトリを実際に public にする(公開する)最終アクション**。
   それ以前の作業(コード整備、ドキュメント作成、ローカルでの検証)は
   自律的に進めてよいが、「世界に公開する」という不可逆的なアクションの
   実行前には、一度状況をまとめて人間に提示すること。
5. `PRINCIPLES.md` の複数の原則が明確に矛盾し、セクション8の優先順位表
   でも解決できないと判断した場合。

上記以外の細かい判断(実装の細部、文言、構成など)は確認を求めず進めること。
確認を求めすぎることは、このプロジェクトの目的(自律的な自己改善)に反する。

## 進捗管理

- リポジトリ直下に `PROGRESS.md` を作成・更新し続けること。フォーマット目安:

  ```
  ## Phase X: <フェーズ名>
  - Status: Not Started / In Progress / Done / Blocked
  - Summary: 何をしたか、何を判断したか(理由も一言)
  - Open Issues: エスカレーション対象があればここに記載
  ```

- 各 `/loop` 反復の開始時に `PROGRESS.md` を読み、直前の状態から再開すること。
  セッションをまたいでも迷わないよう、常に「今どのフェーズか」を明記する。

## ロードマップ(フェーズの目安。詳細タスクは Claude Code が都度分解する)

- **Phase 0: 現状監査**
  現行の SKILL.md・関連ファイルを読み込み、`SPEC.md` セクション1の記述との
  差異を確認する。API キー漏出・個人情報の有無を最初にスキャンする
  (これは最優先。他のフェーズより先に必ず実施する)。

- **Phase 1: セキュリティ・匿名性の確保**
  `SPEC.md` セクション5・8 に基づき、secret scan 導入、`.gitignore` 整備、
  匿名性の確認を行う。エスカレーション条件に該当するものが見つかれば
  ここで止めて報告する。

- **Phase 2: リポジトリ構成の整理**
  `PRINCIPLES.md` を踏まえたモノレポ構成(pipeline / notetype / examples /
  docs)に整理する。

- **Phase 3: 技術方針の確定**
  音声処理(ローカル Whisper の是非)、Anki import 方式の比較・決定を行い、
  `PROGRESS.md` に決定理由を記録する。

- **Phase 4: サンプルコンテンツの整備**
  著作権フリーの教材を探し、`examples/` と `SOURCES.md` を作成する。

- **Phase 5: セットアップ体験の作り込み**
  `install.sh` 等の作成、README の執筆(A4一枚の制約を厳守)。
  Claude Code 自身が「何も知らない第三者」のつもりで再現テストを行う
  (`SPEC.md` セクション7)。

- **Phase 6: ライセンス・帰属の整備**
  `LICENSE`(Apache 2.0)、`NOTICE`(AnKing 帰属)を整備する。

- **Phase 7: 最終確認**
  Definition of Done を満たしているか総点検する。セキュリティ・匿名性・
  ライセンスの最終チェックを行う。

- **Phase 8: 公開(エスカレーション対象)**
  実際にリポジトリを public にする前に、Phase 0〜7 の完了状況を
  `PROGRESS.md` にまとめ、人間に最終確認を仰ぐ。承認後、公開作業を行う。

- **Phase 9: 公開後の運用**
  Issue / PR 対応。`PRINCIPLES.md` 原則6(コミュニティ文化)は Open
  Question のままなので、実例が出てきた時点で `PROGRESS.md` に判断基準の
  叩き台を記録し、蓄積していく。

## 開始時のチェックリスト(毎回の /loop 冒頭で確認)

1. `PRINCIPLES.md` を読む
2. `SPEC.md` を読む
3. `PROGRESS.md` を読み、現在のフェーズを把握する
4. エスカレーション条件に該当する未解決事項がないか確認する
5. 現在のフェーズの次のタスクを自分で分解し、実行する
6. 実行後、`PROGRESS.md` を更新する
