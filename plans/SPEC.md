実装の詳細(ファイル構成、言語、ライブラリ選定など)を自律的に判断してよい。

## 2. 音声処理の方針

- 現行の OpenRouter + Whisper large-v3 を既定路線として維持してよい。
- 「英語音声であればローカルの Whisper でも十分な精度が出るのではないか」
  という仮説がある。これは検証・採用の判断を Claude Code に委ねる。
  判断基準:
  - 精度がAPI版と実用上遜色ないか
  - セットアップの複雑さを増やさないか(`PRINCIPLES.md` 原則4: シンプルさ)
  - API キー不要になることでオンボーディングが容易になるという利点
  - 上記を比較し、デフォルトをどちらにするか、あるいは選択式にするかを
    Claude Code が決めてよい。ただし決定理由は `PROGRESS.md` に残すこと。

## 3. カード生成の方針

- Claude Code + DeepSeek という構成は維持する前提。
- ただし配布先のユーザーは Claude Code や DeepSeek の API キーを持っていない
  可能性が高い。セットアップ手順に「どこでキーを取得し、どこに設定するか」
  を明確に含めること(README 1ページの制約と両立させる工夫が必要)。
- 生成の正確性についての追加のバリデーション機構は原則として追加しない
  (`PRINCIPLES.md` 原則1)。

## 4. Anki への Import 方式

- 現状の「Claude Code 依存かつ環境依存」という状態は、配布物として不安定な
  リスク要因である。以下の選択肢を比較検討し、Claude Code が最適と判断する
  方式を採用してよい:
  - AnkiConnect(Anki アドオン)経由での import に統一する
  - `.apkg` ファイルをエクスポートし、ユーザーが手動でダブルクリック import
    する方式にする(最もシンプルだが自動化度は下がる)
  - OS ごとの分岐処理を用意する
- 判断基準は Definition of Done(セクション7)と `PRINCIPLES.md` 原則2
  (自動化するが人間の理解可能性を残す)。

## 5. セキュリティ要件(最優先の注意事項)

API キーや認証情報が漏出するリスクがある箇所が既知である。以下を必須要件
とする:

- `.env` 等の認証情報ファイルは必ず `.gitignore` に含め、リポジトリ履歴を
  含めて一切コミットされていないことを確認する。
- コミット前の secret scan(例: gitleaks, trufflehog 等)を導入し、
  CI にも組み込む。
- 既存の開発過程で誤ってコミットされたキーがないか、git 履歴全体を
  遡って確認する。見つかった場合:
  1. 直ちに該当キーを無効化・再発行するよう `PROGRESS.md` に明記し、
     人間(プロジェクト管理者)に必ずエスカレーションする(`GOAL.md` の
     エスカレーション条件を参照)。
  2. 履歴からの完全な除去(force-push を伴う書き換え)は破壊的操作のため、
     人間の確認なしに実行しない。
- サンプルコード・README 内に実際のキーやトークンを一切含めない
  (プレースホルダーのみ使用)。
- 個人を特定できる情報(実名、所属大学、個人のメールアドレスなど)が
  コード・コミットメッセージ・コミット履歴・サンプルファイルのメタデータ
  等に残っていないか確認する(匿名公開の前提のため)。

## 7. Definition of Done(受け入れ基準)

このプロジェクトが「公開可能」と判断される最終基準:

> **README を読んだだけで、技術に詳しくない人がゼロから迷わず動かせる状態。
> かつ README は A4 一枚に収まる分量であること。**

これを満たすために:
- README は「3行(またはワンコマンド)で動かす」導入を含む。
- 詳細な説明・トラブルシューティングは README 本体ではなく `docs/` 配下に
  分離し、README の分量を圧迫しない。
- インストールスクリプト(`install.sh` 等)により、依存関係の解決から
  notetype の登録までをできる限り自動化する。
- API キーの取得・設定手順は、A4一枚に収まる範囲で簡潔に案内する
  (詳細は別ドキュメントにリンクする形でよい)。

Claude Code は実装後、「自分が全く事情を知らない第三者になったつもりで
README だけを見て再現できるか」を自己検証すること。

公開後もコミュニティとして、持続可能で、様々な人が開発に参加でき、build 可能な状態であること。
日本語圏、英語圏、双方の人が開発を続けられること。
Anki のように、様々な人が簡単にプラグインを作れるような環境を整え、持続可能にすること。
様々な環境に対応させ、包括的なエコシステムと、モジュール型の設計を採用すること。

開発からコミット、更新までの流れが非常にスムーズであること。
すでにある、SKILLのOSSのベストプラクティスを継承すること。


## 8. 匿名性

- コミット author 情報、README のクレジット表記等はすべて無難なニックネーム
  に統一する。
- 個人が特定されうる痕跡(コメント内の実名、特定の大学固有の内容、
  過去のコミット履歴に残る個人情報)がないか確認する。
